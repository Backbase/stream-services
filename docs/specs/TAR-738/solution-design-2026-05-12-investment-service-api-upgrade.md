## 1. Approach

**Summary:** Update the `investment-service-api.version` property from `1.4.1` to `1.6.0` in `investment-core/pom.xml`, which automatically triggers regeneration of all API clients via the existing `boat-maven-plugin` configuration. Any compilation errors in production or test code caused by API changes are resolved, and the build is verified with `mvn clean install`.

**Decisions:**

| Decision | Why | Rejected Alternatives |
|---|---|---|
| Single property bump at `investment-service-api.version` | The YAML spec filename and artifact coordinates are both version-parameterized; changing one property drives both unpack and codegen automatically | Manually updating artifact coordinates separately — unnecessary duplication |
| Discover breaking changes at compile time | Compiler precisely identifies all usages of changed APIs; no manual spec-diff required | Pre-inspecting 1.6.0 YAML diff (Approach B) — adds overhead and still requires compile verification |
| Fix production code and test code in the same pass | Keeps the build consistently compilable; avoids a broken intermediate state | Fixing production first then tests in a separate commit — creates a non-compilable intermediate state |

**Technical Standards Compliance:**

| Standard | How Satisfied |
|---|---|
| Never hand-edit `target/generated-sources/` (AGENTS.md) | All changes confined to `pom.xml`, production source, and test source; generated sources regenerated by Maven |
| Lombok + MapStruct conventions (AGENTS.md) | No new manual code introduced; existing Lombok/MapStruct patterns preserved |
| Reactive/non-blocking pattern (AGENTS.md) | No reactive pipeline changes; version bump only affects generated client method signatures |

**SLA Compliance:**

| SLA | How Met |
|---|---|
| N/A — no performance-sensitive functionality | No SLA changes required |

## 2. Design

### 2.1 Structure

```
stream-investment/investment-core/pom.xml
    ├── investment-service-api.version: 1.4.1 → 1.6.0       [CHANGED]
    └── boat-maven-plugin (config unchanged)
           ├── Execution 1: generate-webclient-embedded
           │    └── com.backbase.investment.api.service.v1    [REGENERATED]
           └── Execution 2: generate-rest-template-embedded
                └── com.backbase.investment.api.service.sync.v1 [REGENERATED]

Production source (fix if breaking changes found):
    ├── saga/InvestmentSaga.java
    ├── saga/InvestmentAssetUniverseSaga.java
    ├── service/InvestmentAssetUniverseService.java
    ├── service/InvestmentRiskAssessmentService.java
    ├── service/InvestmentClientService.java
    ├── service/InvestmentPortfolioService.java
    ├── service/resttemplate/InvestmentRestAssetUniverseService.java
    ├── configuration/InvestmentClientConfig.java
    └── configuration/InvestmentRestServiceApiConfiguration.java

Test source (fix if breaking changes found):
    └── test/java/**/*Test.java  (all affected test classes)
```

### 2.2 Data Flow

```
1. Developer updates investment-service-api.version to 1.6.0 in pom.xml
2. mvn generate-sources triggers:
   a. maven-dependency-plugin unpacks investment-service-api:1.6.0:zip:api
      → writes investment-service-api-v1.6.0.yaml to target/yaml/
   b. boat-maven-plugin execution 1: generates WebClient API + model classes
      → target/generated-sources/.../com.backbase.investment.api.service.v1/
   c. boat-maven-plugin execution 2: generates RestTemplate API + model classes
      → target/generated-sources/.../com.backbase.investment.api.service.sync.v1/
3. mvn compile identifies source incompatibilities with new generated classes
4. Production code fixed where compiler reports errors
5. mvn test-compile identifies test incompatibilities
6. Test fixtures updated to match 1.6.0 model structure
7. mvn clean install verifies full build passes with no skipped/failed tests
```

### 2.3 State

| State | Location | Why |
|---|---|---|
| API version | `pom.xml` property `investment-service-api.version` | Build-time configuration, not runtime state |
| Generated sources | `target/generated-sources/` (transient) | Rebuilt on every `generate-sources` phase; not committed to VCS |

## 5. Edge Cases

| Scenario | User Sees          | Implementation                 |
| -------- | ------------------ | ------------------------------ |
| Loading  | N/A                | N/A — no runtime loading state in a build upgrade task |
| Empty    | N/A                | 1.6.0 API introduces no breaking changes affecting `investment-core` |
| Error    | Build Failure      | Identify affected files from compiler output and fix incompatibilities |

**Detailed Edge Cases:**

| Scenario | Observation | Implementation Approach |
|---|---|---|
| Error: test fixtures reference removed or renamed model fields | `mvn test-compile` fails with "cannot find symbol" on fluent setter calls | Identify affected test files from compiler output; update fixture construction to use current 1.6.0 field names or remove references to removed fields |
| Error: production code references removed or renamed API methods | `mvn compile` fails with "cannot find symbol" on API client method calls | Identify affected service/saga files from compiler output; replace with equivalent methods from new API |
| Error: new enum values require additional `enumNameMappings` | `boat-maven-plugin` codegen fails during `generate-sources` with enum name collision | Add or update `<enumNameMapping>` entries in both plugin executions in `stream-investment/investment-core/pom.xml` |

## 6. Tests

### 6.1 Scenarios

| Component/Service | Scenarios |
|---|---|
| `investment-core` module build | `mvn clean install` completes without error; zero tests skipped; zero test failures |
| Test fixtures (all `*Test.java` files) | After version bump, all test fixture construction compiles against 1.6.0 model classes |
| Saga tests (`InvestmentSagaTest`, `InvestmentAssetUniverseSagaTest`, `InvestmentContentSagaTest`) | Existing happy-path (`State.COMPLETED`) and error-path (`State.FAILED`) scenarios pass unchanged |
| Service tests (all `*ServiceTest.java` and RestTemplate test classes) | Existing Mockito/StepVerifier scenarios pass with any updated mock return types |
| Configuration tests (`InvestmentClientConfigTest`, `InvestmentServiceConfigurationTest`, `InvestmentRestServiceApiConfigurationTest`) | Spring context wiring tests pass with regenerated API client classes |

### 6.2 Testing Approach

| Decision | Why | Source |
|---|---|---|
| Rely on existing test suite as regression coverage — no new test cases added | The upgrade does not change business logic; the existing suite covers all sagas, services, and configuration; no new functionality is introduced | Analysis Component 8; JIRA task Out of Scope |
| Use JUnit 5 + Mockito + StepVerifier for any fixture corrections | Consistent with established test patterns across all investment-core tests | Analysis Component 8 (Test Conventions) |
| Construct test fixtures programmatically using fluent setters | Module has no `src/test/resources`; all fixtures are built in-code | Analysis Component 7 |
